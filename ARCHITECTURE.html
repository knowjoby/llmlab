1 Executive Summary

Shakespeare GPT is a miniature artificial intelligence that has been taught to write in the style of Shakespeare — one character at a time. It was built from scratch using a type of AI called a "transformer," the same fundamental design that powers ChatGPT, only dramatically smaller. Where ChatGPT has tens of billions of parameters, this model has 216,576 — small enough to train in minutes on a laptop, yet large enough to produce recognisably Shakespearean prose. The entire application runs locally in your browser via a Python web server; nothing is sent to the cloud.

Beyond just generating text, the app is designed as a learning tool. Every decision the model makes during generation is made visible: which characters it was considering, how confident it was, how that confidence changed over time. Separate tabs let you compare how temperature (the model's "boldness dial") changes the output, replay the model's learning process step by step, paste any text to see how well the model would have predicted it, and peer inside the model's attention mechanism to see literally which characters it is "looking at" when deciding what to write next. The project also ships with three AI sub-agents — Architect, Tester, and Reviewer — that can extend, validate, and audit the codebase automatically.

Who this is for

The primary user is a non-technical AI researcher or product owner who wants to understand how large language models work from the inside — not how to build them, but how they behave. Every label in the UI avoids jargon. The Architect agent writes plans in plain English with business analogies. The system runs on a Mac with no GPU required.

What problems it solves

The "black box" problem

Most LLM demos show only the output. This app shows every decision, confidence score, and attention weight — the AI's "thinking" made visible in real time.

The "is it copying?" question

The verbatim detector checks whether generated text was memorised from training data, answering the core copyright / originality question with a character-level overlay.

The "temperature dial" mystery

The temperature gallery generates the same prompt at five boldness levels side by side, making the abstract concept of sampling temperature immediately tangible.

The "how did it learn?" question

The training replay loads model snapshots from different training steps and shows how perplexity dropped and writing quality improved as training progressed.

3 Model Architecture — Technical Deep Dive

Character-level tokenisation

This model operates at the character level — every letter, digit, punctuation mark, newline, and space is a separate token. The vocabulary is built by scanning the entire Shakespeare corpus and collecting the unique characters. There are exactly 65 unique characters in the dataset.

# Built once at module load time from tinyshakespeare.txt
chars      = sorted(set(_text))        # 65 unique chars
VOCAB_SIZE = len(chars)                # 65
stoi       = {c: i for i, c in enumerate(chars)}   # char → int
itos       = {i: c for i, c in enumerate(chars)}   # int → char
encode     = lambda s: [stoi[c] for c in s if c in stoi]
decode     = lambda l: "".join(itos[i] for i in l)

Characters not in the Shakespeare vocabulary (e.g., emoji, non-Latin letters) are silently dropped by the if c in stoi guard in the encoder. The Score tab surfaces a note when this happens.

Embedding layers

Two learned embedding tables convert integers to dense vectors of dimension 64 (N_EMBD = 64). The token embedding maps character ID to a 64-dimensional meaning vector. The positional embedding maps position index (0–127) to a 64-dimensional location vector. Both are added before the first transformer block.

self.tok_emb = nn.Embedding(VOCAB_SIZE, N_EMBD)  # shape: (65, 64)
self.pos_emb = nn.Embedding(BLOCK_SIZE, N_EMBD)  # shape: (128, 64)

# In forward():
pos = torch.arange(T, device=DEVICE)             # [0, 1, ..., T-1]
x   = self.drop(self.tok_emb(idx) + self.pos_emb(pos))

Transformer block structure

The model stacks 4 identical transformer blocks (N_LAYER = 4). Each block uses Pre-LN (LayerNorm before each sub-layer, not after), which is more stable during training than the original transformer design. Both sub-layers use residual connections.

# Block.forward() — Pre-LN with residuals
x = x + self.attn(self.ln1(x))   # residual + attention
x = x + self.mlp(self.ln2(x))    # residual + feed-forward

CausalSelfAttention

This is the core mechanism that lets each position in the sequence "look back" at earlier positions. "Causal" means a position cannot see future positions — enforced by a lower-triangular mask that sets future attention scores to negative infinity before softmax. There are 4 attention heads (N_HEAD = 4), each operating in 16-dimensional space (64 / 4).

# Q, K, V all produced from one Linear in one pass
q, k, v = self.c_attn(x).split(N_EMBD, dim=2)

# Reshape for multi-head: (B, T, n_head, head_dim) → (B, n_head, T, head_dim)
q = q.view(B, T, self.n_head, self.head_dim).transpose(1, 2)

# Scaled dot-product attention
att = (q @ k.transpose(-2, -1)) * (self.head_dim ** -0.5)

# Apply causal mask — positions cannot see the future
att = att.masked_fill(self.bias[:, :, :T, :T] == 0, float("-inf"))
att = F.softmax(att, dim=-1)

# Attention capture hook — set _capture=True before forward pass
if self._capture:
    self._last_attn = att[0].detach().cpu()   # shape: (n_head, T, T)

MLP (feed-forward network)

After attention, each position is processed independently through a two-layer network with a 4x expansion. GELU activation is used instead of ReLU (the standard choice for modern transformers, including GPT models).

self.net = nn.Sequential(
    nn.Linear(N_EMBD, 4 * N_EMBD),   # 64 → 256
    nn.GELU(),
    nn.Linear(4 * N_EMBD, N_EMBD),   # 256 → 64
    nn.Dropout(DROPOUT),
)

Language model head

After the 4 transformer blocks and a final LayerNorm, a single linear projection maps the 64-dimensional hidden state to 65 logits — one per vocabulary character. No bias term is used (matching the GPT-2 design). The softmax of these logits gives the probability distribution over next characters.

self.ln_f = nn.LayerNorm(N_EMBD)
self.head = nn.Linear(N_EMBD, VOCAB_SIZE, bias=False)

# In forward():
return self.head(self.ln_f(x))   # output shape: (B, T, 65)

Weight tying note: The language model head (head) is not weight-tied to tok_emb in this implementation. Weight tying (sharing weights between the input embedding and output projection) is common in larger models to save parameters, but at 216K params it is not necessary here.

Weight initialisation

All linear and embedding weights are initialised from a normal distribution with standard deviation 0.02 (matching the GPT-2 paper). All biases are initialised to zero.

Training flow

When the user clicks "Train from scratch"
tinyshakespeare.txt (1.1M characters, already loaded at startup) │ encode() — stoi lookup — already done at module level ▼ _data = torch.tensor(encode(_text)) ─────────► train_data (90%) / val_data (10%) │ ▼ stream_training(n_steps, lr_val) │ ├─ _build_model(lr) → creates GPT instance + AdamW optimizer, sets _model / _optimizer │ └─ for step in range(1, n_steps+1): │ ├─ get_batch("train") │ │ random start indices (batch_size=32) │ │ x: (32, 128) int tensor — input context │ └ y: (32, 128) int tensor — target (x shifted right by 1) │ ├─ logits = _model(x) shape: (32, 128, 65) │ ├─ loss = F.cross_entropy(logits.view(-1, 65), y.view(-1)) │ scalar — average negative log-prob of correct next char │ ├─ loss.backward() fills .grad on all parameters ├─ clip_grad_norm_(max=1.0) prevents gradient explosion ├─ _optimizer.step() AdamW parameter update │ ├─ if step in {100,250,500,750,1000,1500,2000,3000,4000,5000}: │ save_checkpoint(step) → checkpoints/step_NNNNN.pt │ └─ if step % 200 == 0: estimate_loss() (20 batches, no_grad) → train/val loss yield log, status, make_loss_plot() → streams to browser └─ After loop: torch.save(_model.state_dict(), "tiny_shakespeare.pt") _model.eval() yield final log, model_status_str(), final chart

Attention capture flow

Inside show_attention() — extracting what the model is "looking at"
context_str (user input, last 18 chars used) │ context_str = context_str[-18:] ▼ tokens = encode(context_str) list of ints idx_in = torch.tensor([tokens]) shape: (1, T) │ ├─ for block in _model.blocks: │ block.attn._capture = True ← arm the capture flag │ ├─ with torch.no_grad(): │ _model(idx_in) ← forward pass triggers capture │ CausalSelfAttention.forward(): │ att = softmax(Q@K^T / sqrt(d)) │ if self._capture: │ self._last_attn = att[0].detach().cpu() shape: (4, T, T) │ ├─ for block in _model.blocks: │ block.attn._capture = False ← disarm (must always reset) │ ├─ attn_layers = [block.attn._last_attn for block in _model.blocks] │ list of 4 tensors, each shape: (n_head=4, T, T) │ ├─ avg_all = mean over layers of attn.mean(heads) shape: (T, T) │ ├─ last_row = avg_all[-1] shape: (T,) │ = how much each previous char was attended to │ when predicting the NEXT token │ ├─ make_attention_strip_html(chars, last_row) → HTML bar chart └─ make_attention_chart(chars, attn_layers) → matplotlib Figure
Always reset _capture after the forward pass. If an exception occurs between setting _capture = True and resetting it to False, subsequent generate calls will continue writing to _last_attn, wasting memory and potentially returning stale attention data. The current implementation uses a sequential reset (not a try/finally block) — a future improvement would add error safety here.

All shared mutable state lives as module-level variables in app.py. There is no class wrapping them. Gradio handler functions access them via Python's global keyword. This is a deliberate trade-off — see Section 7 for rationale.

7 Key Design Decisions and Trade-offs

Decision Chosen approach Why Trade-off
Character-level vs. subword tokenisation Character-level: one token per character, vocab size 65 Makes every decision visible and explainable. A non-technical user can see "the model picked the letter 'R'" without needing to understand BPE tokens. Shakespeare's vocabulary is also small enough that character-level is viable. Much slower to generate meaningful output than word-piece tokenisers. The model must learn spelling from scratch. Context window of 128 chars covers only a few sentences rather than paragraphs.
Single-file architecture (app.py ~1,066 lines) Everything in one Python file: model classes, data loading, all handlers, all helpers, Gradio UI wiring Eliminates import complexity and relative path issues. Any developer can open one file and understand the entire system. AI agents (who read files) see the complete picture in a single read. At >1,000 lines the file becomes harder to navigate. Adding further features risks making it unwieldy. A future refactor might split into model.py, handlers.py, helpers.py.
Global state vs. class-based state Module-level variables (_model, _gen_history, etc.) Gradio handler functions need to share state across button clicks without passing it through Gradio's gr.State component (which serialises to JSON and cannot hold PyTorch tensors or model objects). Module-level globals are the simplest solution that actually works. Not thread-safe for concurrent users. A single Gradio server process serves all users with the same model. Acceptable for a single-user local app; would require complete redesign for a shared deployment.
Gradio 6 streaming generators Python generator functions (yield) for all long-running tabs Gradio 6 natively converts generator functions to streaming WebSocket responses. The browser receives partial updates in real time. No polling, no explicit async code, no threading required from the developer. Every yield must emit a tuple of exactly the right length. Using gr.skip() as placeholder is necessary when skipping updates. Generators cannot be interrupted mid-stream from the UI without a stop button (not yet implemented).
Matplotlib figures returned directly gr.Plot accepts a matplotlib.figure.Figure object directly Simpler than converting to base64 PNG strings. Gradio handles the serialisation. Dark theme is applied directly to the figure via _dark_ax() before returning. Matplotlib colors must be in #rrggbb hex format. The string format rgb(r,g,b) silently fails or throws errors in Matplotlib's color parser. All color values in the app use hex (e.g., #f78c6c not rgb(247,140,108)).
N-gram verbatim detection Python set of all 6-character substrings from training text O(1) lookup per n-gram. Conceptually transparent — a match means "this exact 6-char string appears in the training data." No model inference needed. Startup cost (~1-2 seconds) is paid once. N-gram matching is not semantic. It cannot detect paraphrasing or near-duplicate passages. A 6-gram match may be coincidental (common phrases like "the" appear everywhere). Embedding similarity would be more accurate but requires a second inference pass per generation.
Attention capture via instance flag self._capture = False and self._last_attn = None on each CausalSelfAttention instance Avoids modifying the forward() method signature. Existing generate calls that don't need attention data incur zero overhead. The flag is toggled only for the Attention tab's one-off analysis pass. The flag is shared state on the module — not thread-safe. Requires careful reset after each use. A future improvement: use PyTorch's register_forward_hook() API which is designed for this pattern and is thread-safe.
Checkpoint zero-padding (step_00100.pt) 5-digit zero-padded step number in filename Ensures that lexicographic sort (sorted(os.listdir(...))) produces the correct chronological order. Without padding, step_1000.pt would sort before step_250.pt because "1" < "2". Slightly longer filenames. Step numbers above 99,999 would overflow — acceptable since training runs top out at 5,000 steps.

10 Improvement and Extension Guide

The most common change to this codebase is adding a new tab. Here is the complete sequence, in order. Each step has a concrete instruction and a potential failure mode to watch for.

How to add a new tab — complete checklist

1
Write the handler function

Decide first: is this tab long-running (use a generator with yield) or instant (use return)? Generator functions can stream updates to the browser. Instant functions block until complete.

# Generator pattern (for long operations)
def my_new_handler(input_a, input_b):
    if _model is None:                          # ← REQUIRED guard
        yield "⚠️  No model loaded.", None
        return
    # ... do work ...
    yield progress_text, chart_figure          # ← must match outputs= list

# Instant return pattern (for fast operations)
def my_new_handler(input_a):
    if _model is None:
        return "⚠️  No model loaded.", None
    # ... do work ...
    return html_string, figure

Watch out: The tuple length of every yield (or the return) must exactly match the number of items in the outputs=[] list you wire in Step 3. A mismatch causes silent failures.

2
Add the UI layout inside a with gr.Tab() block

Insert the block at the correct position in the with gr.Blocks() as demo: section. Tabs appear in the order they are defined. Follow the existing pattern: left column for inputs, right column (wider) for outputs.

with gr.Tab("🆕  My New Tab"):
    gr.Markdown("Plain English description of what this tab does.")
    with gr.Row():
        with gr.Column(scale=1):
            my_input   = gr.Textbox(label="Your input", lines=3)
            my_slider  = gr.Slider(1, 100, value=50, label="A setting")
            btn_my_tab = gr.Button("▶️  Run it", variant="primary")
        with gr.Column(scale=2):
            my_output_html  = gr.HTML(label="Results")
            my_output_chart = gr.Plot(label="Chart")

UI rules: All labels must be plain English — no variable names as labels. No jargon. Use the existing temperature/label vocabulary where possible ("Focused", "Creative", etc.).

3
Wire the button to the handler with .click()

Add the event binding in the "Wire up events" section at the bottom of the with gr.Blocks(): block.

btn_my_tab.click(
    fn=my_new_handler,
    inputs=[my_input, my_slider],
    outputs=[my_output_html, my_output_chart]
)

Watch out: The inputs=[] list must match the handler's function signature exactly (same order, same number). The outputs=[] list must match the number of items yielded/returned.

4
Add the if _model is None guard

Every handler that uses _model must start with this guard. Without it, the handler will crash with an AttributeError if the user hasn't loaded a model yet.

# For generators:
if _model is None:
    yield "<p style='color:#f87171'>⚠️  No model loaded. Go to Train tab first.</p>", None
    return

# For instant return functions:
if _model is None:
    return "<p style='color:#f87171'>⚠️  No model loaded.</p>", None

Common pitfalls to avoid

PitfallSymptomFix
Wrong yield tuple length Gradio silently ignores updates or throws a runtime error Count the items in outputs=[] and ensure every yield has exactly that many values. Use gr.skip() for outputs you don't want to update on that yield.
RGB color string in Matplotlib Plot renders with wrong colors or throws ValueError: Invalid RGBA argument Replace all rgb(r,g,b) with #rrggbb hex strings.
Missing model guard AttributeError: 'NoneType' object has no attribute 'eval' Add if _model is None: yield/return error, None at the very start of the handler.
Calling inference without no_grad Memory leak — gradient buffers accumulate across calls Wrap all _model() calls in with torch.no_grad(): except during training.
Forgetting to call _model.eval() Dropout is active during inference — outputs are stochastic in a bad way Call _model.eval() before any inference pass. Call _model.train() before training.
Not resetting attention capture flag Future generate calls contaminate _last_attn with stale values Always reset block.attn._capture = False after the forward pass, ideally in a try/finally block.
Checkpoint file without zero-padding Replay tab shows checkpoints out of chronological order Always use f"step_{step:05d}.pt" (five digits, zero-padded).
New tab uses gr.Blocks(theme=...) Theme/CSS ignored, Gradio throws deprecation warning Theme and CSS must be in demo.launch(theme=..., css=...).

Ideas for future extensions

Stop button for generators

Add a "Stop generation" button. Gradio supports this via a cancels= parameter on the event. Currently, the only way to stop a long generation is to reload the page.

Perplexity comparison tab

A tab that accepts two text samples and compares their perplexity scores — useful for understanding which text is more "Shakespearean" to the model.

Top-k vs. top-p sampling

Add nucleus sampling (top-p) as an alternative to top-k. Expose it as a toggle in the Generate tab. A slider for p (0.1–1.0) would replace the top-k slider when enabled.

Model size comparison

A tab that trains models of different sizes (1 layer, 2 layers, 4 layers) and compares their perplexity curves side by side — demonstrating the scaling law in miniature.

Register forward hooks (safety)

Replace the _capture instance flag pattern with PyTorch's register_forward_hook() API. Safer, thread-compatible, and does not require resetting after each use.

Weight tying

Share weights between tok_emb and head (set head.weight = tok_emb.weight). This reduces parameters by 4,160 and often improves perplexity, especially for small vocabularies.
